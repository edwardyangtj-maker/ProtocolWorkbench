# 协议联调工作台需求与架构设计说明 v1.0

## 1. 项目背景

当前项目中，后端由我负责开发，上位机设备端由同事负责开发。双方在协议对接过程中经常围绕字段定义、消息类型、响应规则、ACK、心跳、TCP 分帧方式、JSON 字段类型、接口触发顺序等问题产生争议。现有方式主要依赖手写 JSON、临时 Socket 工具、口头确认和日志排查，缺少一个能够把协议定义、报文模板、通信端点、场景流程和运行日志统一管理的工具。

因此，本项目计划开发一个桌面端“协议联调工作台”，用于后端、上位机设备端、前端接口以及其他通信模块之间的协议测试、接口验证、场景仿真和问题复现。该工具应支持 TCP、HTTP、UDP、WebSocket 等当前项目实际使用的通信方式，并以 JSON 报文为第一阶段重点，同时保留对 Text、Raw、Hex、Binary 等数据格式的扩展能力。

该工具的核心价值不是简单发包，而是将协议变成可视化、可配置、可执行、可复现、可导出的测试资产。后续我可以在工具中创建环境、配置端点、设计报文模板、配置 ACK/Response/Heartbeat、编排测试场景、并行启动多个环境或端点，并将项目配置导出为 JSON 配置包交给同事加载使用，从而减少协议争议，提高联调效率。

## 2. 项目定位

本项目定位为一个桌面端协议联调与通信仿真工具。它既可以作为 TCP/HTTP/UDP/WebSocket 调试器使用，也可以作为后端、设备端、上位机之间的协议验证平台使用。第一版重点解决当前项目中的 TCP + JSON、HTTP + JSON、UDP + JSON、WebSocket + JSON 联调需求，同时支持自定义 TCP 分帧和报文模板持久化。

更准确地说，本工具不是单纯的“Socket 调试助手”，而是一个“协议编排 + 通信仿真 + 模板管理 + 日志追踪 + 配置导出”的桌面工作台。

第一版目标是快速可用，不追求完整商用化，不做数据库、不做云端协作、不做复杂权限管理、不做复杂流程图拖拽。所有配置优先使用本地 JSON 文件持久化，便于快速开发、快速交付、快速给同事导入使用。

## 3. 当前版本总体目标

第一版需要实现以下总体目标：

1. 支持 TCP、HTTP、UDP、WebSocket 四类通信方式。
2. 支持每个环境、每个端点独立启动、停止、发送和接收。
3. 支持基于 TCP 的自定义 JSON 协议解析，例如 Start-End、Delimiter、Length-Prefix 等分帧方式。
4. 支持 JSON 报文的源码编辑和树形编辑两种模式。
5. 支持消息模板、ACK 模板、Response 模板、Heartbeat 模板持久化。
6. 支持单包发送、只接收、接收后自动回复、发送后等待 Response、是否需要 ACK、心跳模式等测试方式。
7. 支持多个环境或多个场景并行执行。
8. 支持实时日志窗口和本地日志文件，日志需要包含原始文本、原始 Hex、解析后 JSON、端点、方向、时间、匹配模板和错误信息。
9. 支持项目配置导入和导出，导出的配置包可交给同事加载后复现相同测试。
10. 保持架构可扩展，后续可继续扩展 XML、Binary、Protobuf、串口、MQTT、压力测试和自动化测试报告。

## 4. 核心概念定义

### 4.1 Project 项目

Project 表示一个完整的协议联调工程。一个项目中包含环境、端点、分帧规则、消息模板、场景、变量和日志配置。项目可以保存到本地目录，也可以导出为 ZIP 包交给其他人导入。

### 4.2 Environment 环境

Environment 表示一个联调环境。例如“后端测试环境”“设备模拟环境”“上位机测试环境”“HTTP 接口环境”。每个环境可以绑定多个端点，并拥有自己的环境变量、日志目录、默认模板和启动策略。

环境不是协议本身，而是一组端点和配置的集合。启动一个环境时，可以自动启动该环境下配置的 TCP Server、UDP Endpoint、HTTP Server、WebSocket Server 或其他端点。

### 4.3 Endpoint 通信端点

Endpoint 是真正负责通信的对象。端点可以是 TCP Client、TCP Server、HTTP Client、HTTP Server、UDP Endpoint、WebSocket Client 或 WebSocket Server。每个端点拥有自己的 IP、端口、超时、分帧规则、Payload 类型、心跳配置和日志配置。

端点必须能够独立启动和停止，也必须能够被环境或场景统一调度。

### 4.4 FrameRule 分帧规则

FrameRule 用于解决 TCP 等字节流协议中的消息边界问题。标准 HTTP 和 WebSocket 天然有消息边界，但 TCP 是字节流，没有天然边界，因此必须配置分帧规则。第一版需要支持 Raw、Delimiter、Start-End、Length-Prefix 四种分帧模式。

### 4.5 PayloadCodec 数据编码器

PayloadCodec 用于处理报文内容的编码和解码。第一版重点支持 JSON，同时支持 Text 和 Hex。后续可以扩展 Binary、XML、Protobuf 等。

### 4.6 MessageTemplate 消息模板

MessageTemplate 表示一个可复用的报文模板。模板可以是普通消息、命令消息、查询消息、上报消息、响应消息、ACK 消息或心跳消息。模板中可以使用变量，例如 ${uuid}、${now_ms}、${random_int}，运行时由变量引擎替换为真实值。

### 4.7 Scenario 场景

Scenario 表示一组可执行步骤，用于模拟完整通信流程。例如启动 TCP Server、等待设备上报、校验 JSON 字段、自动返回响应、发送心跳、记录日志等。第一版不做复杂流程图拖拽，采用步骤表格形式实现。

### 4.8 LogRecord 日志记录

LogRecord 表示一次运行事件或一次报文收发记录。日志必须保存原始数据和解析结果，便于定位协议争议。尤其是 TCP 分帧、换行符、起止符、隐藏字符、字段类型错误等问题，必须依赖完整日志证明。

## 5. 功能需求

### 5.1 项目管理

系统需要支持新建项目、打开项目、保存项目、导入项目包和导出项目包。项目配置使用本地 JSON 文件持久化，不使用数据库。导出的项目包应包含环境配置、端点配置、分帧规则、消息模板、场景配置和变量配置。

推荐项目目录结构如下：

```text
MyProtocolProject/
├── project.json
├── environments.json
├── endpoints.json
├── frame_rules.json
├── message_templates.json
├── scenarios.json
├── variables.json
└── logs/
```

### 5.2 环境管理

系统需要支持创建多个环境。每个环境可以配置名称、描述、环境变量、绑定端点、日志目录、是否自动启动端点、是否允许并行运行等信息。

示例环境包括：

```text
后端测试环境
设备模拟环境
上位机测试环境
HTTP 接口环境
WebSocket 测试环境
UDP 通信环境
```

每个环境可以独立启动和停止。环境启动时，应根据配置启动其绑定的端点；环境停止时，应关闭其绑定端点并释放端口。

### 5.3 端点管理

系统需要支持以下端点类型：

```text
TCP Client
TCP Server
HTTP Client
HTTP Server
UDP Endpoint
WebSocket Client
WebSocket Server
```

每个端点需要配置：

```text
端点名称
端点类型
本地 IP
本地端口
目标 IP
目标端口
请求路径或连接地址
Payload 类型
FrameRule 分帧规则
连接超时
读写超时
是否自动重连
是否启用心跳
日志配置
```

TCP Server、HTTP Server、WebSocket Server、UDP Endpoint 需要能够监听本地端口。TCP Client、HTTP Client、WebSocket Client 需要能够主动连接或请求目标地址。

### 5.4 协议与分帧配置

系统需要区分标准协议和自定义协议。

标准协议包括 HTTP、WebSocket、UDP Datagram 等，它们在传输层或应用层已有基本边界。但它们内部仍可承载自定义 JSON 业务消息。

自定义协议主要指 TCP 字节流上承载的自定义消息格式，例如：

```text
*{json}/
{json}\n
4 字节长度头 + JSON Body
固定头 + 长度字段 + Body + 校验字段
```

第一版需要支持以下分帧模式：

1. Raw：不做分帧，直接发送或接收原始内容。
2. Delimiter：根据分隔符拆包，例如 \n、\r\n、END。
3. Start-End：根据起始符和结束符识别完整包，例如 * 开始、/ 结束。
4. Length-Prefix：根据长度字段读取完整包，例如前 4 字节表示 Body 长度。

### 5.5 JSON 编辑器

JSON 编辑器需要支持两种模式。

第一种是源码模式，允许用户直接输入 JSON。源码模式需要支持格式化、压缩、校验、错误提示和保存为模板。错误提示应尽量指出行号、列号和错误原因。

第二种是树形模式，允许用户通过树结构编辑 JSON 字段。树形模式需要包含 Key、Value、Type、Enabled、Description 等列。Type 至少支持 string、number、boolean、null、object、array、raw、expression。

树形模式和源码模式需要能够互相同步。源码解析成功后生成树形结构；树形结构修改后可以重新生成 JSON 源码。源码错误时不允许同步到树形结构。

### 5.6 模板管理

系统需要支持模板设计和模板持久化。模板类型包括：

```text
普通消息模板
命令模板
查询模板
上报模板
响应模板
ACK 模板
心跳模板
HTTP 请求模板
WebSocket 消息模板
UDP 消息模板
TCP 消息模板
```

模板应保存到 message_templates.json 中，也可以按类型拆分为多个文件。模板可以被复制、修改、删除、导出和导入。

模板内容中需要支持变量，例如：

```text
${uuid}
${now_ms}
${timestamp}
${random_int}
${env.backend_host}
${received.header.msgId}
```

运行时需要由变量引擎替换为真实值。

### 5.7 发送、接收与响应模式

系统需要支持以下测试模式：

1. 只发送：选择端点和模板后直接发送报文。
2. 只接收：启动端点监听数据，只记录接收结果。
3. 发送后等待 Response：发送请求后，根据匹配规则等待响应。
4. 接收后自动 Response：收到指定报文后，自动匹配并返回响应模板。
5. 是否需要 ACK：可以配置发送后是否等待 ACK，也可以配置收到消息后是否自动 ACK。
6. 心跳模式：支持主动心跳和被动心跳响应。

ACK、Response、Heartbeat 必须区分：ACK 表示已收到；Response 表示业务处理结果；Heartbeat 表示连接保活。

### 5.8 响应匹配规则

系统需要支持基于 JSONPath 风格的字段匹配。第一版可以使用简化路径，例如：

```text
$.header.msgType == response
$.body.theme == laser_setpower
$.body.result == success
```

匹配操作符至少支持：

```text
eq
ne
contains
exists
regex
```

发送后等待响应时，应支持超时时间、失败动作、重试次数和日志记录。

### 5.9 ACK 与心跳

ACK 模板应允许配置：

```text
是否启用 ACK
ACK 模板
ACK 超时时间
ACK 匹配规则
ACK 失败后的动作
是否自动回复 ACK
```

心跳配置应允许配置：

```text
是否启用心跳
主动发送或被动响应
心跳间隔
心跳模板
心跳响应模板
心跳超时时间
连续失败次数
失败后动作：继续、断开、重连、停止环境、停止场景
```

### 5.10 场景编排

第一版采用步骤表格方式实现场景编排，不做拖拽式流程图。场景步骤包括：

```text
start_endpoint
stop_endpoint
send_message
wait_message
auto_reply
delay
assert_json
loop
```

一个场景可以表示完整通信流程，例如：

```text
1. 启动 TCP Server 9000
2. 等待设备连接
3. 等待 report_scan_img 上报
4. 校验 header.msgType = report
5. 自动返回 response_success
6. 延迟 1000ms
7. 停止端点
```

系统需要支持多个场景并行执行。每个场景的执行状态、每一步结果、错误原因和报文记录都需要写入日志。

### 5.11 日志系统

日志是第一版必须重点实现的模块。系统需要同时提供界面实时日志和本地持久化日志。

界面日志至少分为：

```text
运行日志
发送日志
接收日志
错误日志
原始报文
```

本地日志建议每次运行生成一个目录：

```text
storage/logs/
└── 2026-05-07_后端环境/
    ├── runtime.log
    ├── tx.log
    ├── rx.log
    ├── error.log
    └── packets.jsonl
```

packets.jsonl 是最重要的报文日志文件。每一行保存一条报文记录，至少包含：

```text
时间
环境
端点
方向
协议类型
远端地址
模板名称
原始文本
原始 Hex
解析后 JSON
分帧规则
匹配规则
耗时
状态
错误信息
```

### 5.12 导入导出

系统需要支持项目配置导入导出。导出的配置包应包含所有必要配置，使同事导入后可以复现相同环境和测试流程。

导出内容至少包括：

```text
project.json
environments.json
endpoints.json
frame_rules.json
message_templates.json
scenarios.json
variables.json
README.md
```

日志可选择是否随项目包一起导出。

## 6. 非功能需求

第一版优先满足快速开发、快速使用、配置可迁移、日志可追溯、架构可扩展。界面美观不是第一优先级，但布局必须清晰，避免影响使用。

系统需要支持多个端点并行运行，不能因为一个端点阻塞导致整个界面卡死。网络通信建议使用异步模型或独立任务调度。

系统需要尽量避免写死业务字段。msgType、theme、body、header 等字段应通过模板和匹配规则配置，而不是在代码中固定死。

系统需要具备良好的错误处理能力。JSON 格式错误、端口占用、连接超时、分帧失败、Payload 解码失败、Response 超时、ACK 超时等情况都必须清晰记录并显示。

## 7. 推荐技术方案

最快开发方式建议采用 Python + PySide6 桌面端。

推荐技术栈：

```text
Python 3
PySide6
asyncio
qasync
aiohttp
websockets
标准 socket / asyncio Datagram
本地 JSON 文件持久化
```

选择 Python + PySide6 的原因是开发速度快、JSON 和日志处理方便、异步通信生态成熟、工具类软件更容易快速成型。Qt C++ 的稳定性和长期维护性较好，但第一版开发成本更高，尤其是 JSON 树编辑器、模板系统、场景编排和异步多协议并行运行等部分。

如果后续需求稳定，并且工具需要长期产品化，可以再考虑用 Qt C++ 重构。但第一版建议优先快速可用。

## 11. 关键运行流程

### 11.1 启动环境流程

```text
用户选择环境并点击启动
→ RuntimeManager 读取环境配置
→ 找到环境绑定的 Endpoint
→ 根据 Endpoint 类型创建对应 TransportAdapter
→ 加载对应 FrameRule 和 PayloadCodec
→ 启动端点监听或连接
→ LoggerService 写入启动日志
→ UI 更新端点状态
```

### 11.2 发送消息流程

```text
用户选择端点和消息模板
→ TemplateEngine 渲染变量
→ PayloadCodec 将 JSON/Text/Hex 编码为 bytes
→ FrameCodec 按规则封包
→ TransportAdapter 发送 bytes
→ LoggerService 记录 TX 日志
→ 如果启用 ACK 或 Response，则进入等待匹配流程
```

### 11.3 接收消息流程

```text
TransportAdapter 收到 bytes
→ FrameCodec feed 数据并拆出完整帧
→ PayloadCodec 解码 Payload
→ LoggerService 记录 RX 日志
→ ResponseMatcher 匹配模板或响应规则
→ 如果命中自动响应规则，则生成响应模板并发送
→ UI 显示解析结果和匹配结果
```

### 11.4 场景执行流程

```text
用户启动场景
→ ScenarioRunner 读取步骤表
→ StepExecutor 按顺序执行步骤
→ 每一步调用 RuntimeManager、TransportAdapter、TemplateEngine 或 ResponseMatcher
→ 每一步结果写入日志
→ 如果失败，根据 stopPolicy 决定继续或停止
→ 场景结束后生成执行结果
```

## 13. 开发优先级

第一阶段先搭建项目骨架、数据模型、配置持久化和主界面。第二阶段实现 TCP Client、TCP Server、分帧、JSON 解码和日志系统。第三阶段实现 JSON 编辑器和模板管理。第四阶段实现 HTTP、UDP、WebSocket。第五阶段实现 ACK、Response、Heartbeat、自动响应和响应匹配。第六阶段实现场景步骤表和并行执行。第七阶段实现导入导出和基础使用文档。

最优先完成的是 TCP + JSON + 自定义分帧 + 日志，因为这是当前你和同事协议争议中最关键、最容易立刻产生价值的部分。

## 14. 关键架构原则

第一，传输协议和业务协议必须解耦。TCP、HTTP、UDP、WebSocket 只负责通信通道；JSON 中的 msgType、theme、command、report、response 才是业务协议内容。

第二，TCP 收发和 JSON 解析必须解耦。TCP 只处理 bytes，FrameCodec 负责拆包，PayloadCodec 负责 JSON/Text/Hex 解码。

第三，ACK、Response、Heartbeat 必须分开。ACK 表示收到，Response 表示业务处理结果，Heartbeat 表示保活。三者不能混用。

第四，模板和运行时数据必须分开。模板里可以使用变量，运行时由变量引擎生成真实 msgId、timestamp 等值。

第五，日志必须保留原始字节。只保存 JSON 不够，必须保存 rawText 和 rawHex，否则无法排查分隔符、换行符、编码、包头、隐藏字符和长度字段问题。

第六，第一版不要追求所有协议全覆盖，而要保证当前 TCP、HTTP、UDP、WebSocket + JSON 的主流程稳定可用。

## 15. 主要风险

第一，范围膨胀风险。如果第一版同时做数据库、流程图、脚本插件、压力测试和二进制结构体编辑，会导致开发周期失控。必须先锁定 MVP。

第二，协议抽象混乱风险。如果把 TCP、JSON、业务字段写死在同一个模块，后期扩展 HTTP、UDP、WebSocket 会变得困难。因此必须坚持 Transport、Frame、Payload、Template、Scenario 分层。

第三，日志不足风险。如果日志只显示解析后的 JSON，而不显示原始 Hex 和原始 Text，后续协议争议仍然无法彻底解决。

第四，并行运行风险。多个环境和端点同时启动时，需要避免 UI 阻塞、端口冲突、任务泄漏和异常未释放端口。

第五，字段类型风险。JSON 中字符串、数字、布尔值、null 必须严格区分。UI 不应简单使用“是否加双引号”，而应使用 Type 字段控制。

## 16. 当前版本总结

当前版本的需求可以总结为：开发一个 Python + PySide6 桌面端协议联调工作台，支持 TCP、HTTP、UDP、WebSocket 四类端点的独立启动和并行运行；支持 JSON 报文源码编辑与树形编辑；支持 Raw、Delimiter、Start-End、Length-Prefix 四种分帧规则；支持消息模板、ACK 模板、Response 模板、Heartbeat 模板的本地 JSON 持久化；支持单包发送、监听接收、自动响应、发送后等待响应；支持实时日志窗口和本地 packets.jsonl 持久化日志；支持项目配置导入导出，用于后端、上位机和设备端之间的协议联调、接口验证和问题复现。

第一版的核心目标不是一次性做成全功能商业平台，而是先把当前项目中最容易发生争议的协议结构、报文内容、分帧规则、响应机制和日志证据固定下来。只要这个工具能做到“配置可保存、报文可执行、日志可追踪、问题可复现、配置可交付”，就已经能有效解决当前联调过程中的核心矛盾。
